# DjangoPlay — Helpdesk → IssueTracker Convergence

**Branch**

```
feature/helpdesk-sync-with-issuetracker-1
```

---

# Problem Statement

DjangoPlay currently maintains **two separate issue reporting systems**.

---

## 1️⃣ Helpdesk System

Used for:

* Bug reports
* Support tickets

Core entities:

```
BugReport
SupportTicket
FileUpload
```

Features include:

* throttling
* email notifications
* attachments
* audit logs
* admin UI
* frontend forms

Example flows include the bug reporting modal and support form.

---

## 2️⃣ GenericIssueTracker Library

A reusable third-party issue management system used in DjangoPlay.

Provides:

```
Issue
IssueComment
IssueAttachment
Label
IssueStatusHistory
```

Capabilities:

* lifecycle management
* attachments
* identity snapshots
* DRF API
* UI integration
* signal hooks

---

# Current Problems

Maintaining **two parallel systems causes multiple issues**.

| Problem | Impact |
|--------|-------|
| Duplicate data models | Increased maintenance |
| Separate attachment systems | File duplication |
| Separate UIs | Inconsistent UX |
| Separate APIs | Integration complexity |
| Separate lifecycle rules | Workflow inconsistency |
| Duplicate audit logging | Compliance difficulty |

---

# Goal

Unify the **Helpdesk system** with the **GenericIssueTracker** so that:

```
All Bugs + Support Tickets → become Issues
```

without breaking:

* historical data
* audit trails
* UI flows
* email notifications
* throttling policies

---

# High Level Target Architecture

Current

```
Helpdesk App
   BugReport
   SupportTicket
   FileUpload
```

Future

```
GenericIssueTracker
    Issue
    IssueComment
    IssueAttachment
    Label
```

with a **DjangoPlay integration layer**.

# Design Principles

### 1️⃣ DRY

Avoid duplicate models.

```
BugReport → Issue
SupportTicket → Issue
```

---

### 2️⃣ Library First

GenericIssueTracker becomes the **single source of truth**.

Helpdesk becomes a **thin adapter layer**.

---

### 3️⃣ Preserve Public APIs

Existing endpoints remain functional.

```
/report-bug/
/support/
```

Internally they create **Issues**.

---

### 4️⃣ Maintain Historical Data

Existing helpdesk records must remain accessible.

No data loss allowed.

---

### 5️⃣ Follow Standard Django Architecture

Use:

* service layer pattern
* signals for integrations
* thin views
* reusable serializers
* modular UI services

---

# Scope of This Release

This release performs the **initial convergence layer**.

Helpdesk becomes a **legacy compatibility adapter**.

# 1️⃣ Map Helpdesk Entities to Issues

## BugReport → Issue

| BugReport Field | Issue Field |
|---|---|
| bug_number | metadata |
| summary | title |
| steps_to_reproduce + actual_result | description |
| reporter.email | reporter_email |
| reporter.id | reporter_user_id |
| severity | priority |
| status | status |

---

## SupportTicket → Issue

| Support Field | Issue Field |
|---|---|
| ticket_number | metadata |
| subject | title |
| message | description |
| email | reporter_email |
| user.id | reporter_user_id |
| severity | priority |
| status | status |

```
issue.metadata = {

# 6️⃣ Throttling Integration

```

---

# 7️⃣ Role Visibility Rules

Visibility is controlled using the **Issue model**.

```
Issue.is_public
```

Rules:

| Source | is_public |
|------|------|
| Host App Bug | False |
| Issue UI Bug | True |
| UI Bug with Internal selected | False |

---

# 8️⃣ Canonical Bug Labels

Two canonical labels are introduced.

```
slug: bug-internal
name: 🐞 INTERNAL
color: #ef4444
```

```
slug: bug-public
name: 🐞 PUBLIC
color: #10b981
```

Important rule:

**Labels must never determine visibility.**

Visibility is determined only by:

```
Issue.is_public
```

Labels exist purely for **UI classification**.

---

# 9️⃣ Lifecycle Rules

Bug lifecycle mapping.

| Bug Status | Issue Status |
|---|---|
| NEW | OPEN |
| TRIAGED | OPEN |
| IN_PROGRESS | IN_PROGRESS |
| FIXED | RESOLVED |
| VERIFIED | RESOLVED |
| CLOSED | CLOSED |

Support tickets follow the same lifecycle.

---

# 🔟 Data Migration

Migration command:

```
python manage.py migrate_helpdesk_to_issues
```

Steps:

1️⃣ iterate BugReport  
2️⃣ create Issue  
3️⃣ attach files  
4️⃣ copy metadata  

Then:

5️⃣ iterate SupportTicket  
6️⃣ create Issue  

Migration is **idempotent**.

---

# Label Migration

Existing issues will receive canonical labels.

Example migration script:

```
for issue in Issue.objects.filter(labels__slug="bug"):
    if issue.is_public:
        issue.labels.add(Label.objects.get(slug="bug-public"))
    else:
        issue.labels.add(Label.objects.get(slug="bug-internal"))
```

Remove legacy:

```
support
```

# IssueMutationService Label Assignment

After creating the issue:

```
issue = serializer.save()

if issue.is_public:
    label_slug = "bug-public"
else:
    label_slug = "bug-internal"

label = Label.objects.get(slug=label_slug)
issue.labels.add(label)
```
